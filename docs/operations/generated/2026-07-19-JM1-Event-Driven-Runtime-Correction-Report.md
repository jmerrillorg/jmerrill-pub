## Operating Correction

This is not a new requirement. It restores the original operating design:

> Author approval closes the current gate and automatically triggers the next authorized stage.

The runtime ownership audit proved the important distinction: a stage record, dashboard card, package, or transition log is not the same as a commissioned production executor. Where a title still needs a Cody instruction to begin or complete ordinary editorial work, that process is a Cody-assisted bridge and must remain visibly classified that way until a permanent runtime replaces it.

## Correct Autonomous Editorial Loop

The permanent editorial pipeline must execute in this order:

1. Author approves current package.
2. Approval is validated and recorded.
3. Current gate closes.
4. Title advances to the next governed stage.
5. Next-stage runtime starts automatically.
6. Manuscript is processed.
7. Automated and policy-based QA runs.
8. Exceptions route to Publisher Today.
9. Approved output package is generated.
10. Human-cadence release timer begins.
11. Package is released to the author.
12. Author notification is sent.
13. Next author-review gate opens.

Cody should enter the loop only for runtime failure, exception remediation, publisher-judgment support, uncommissioned capability bridge work, or engineering/deployment work.

## Cadence Policy

Execution starts immediately after authorization. Cadence controls author-facing delivery, not internal execution.

The cadence must be configurable and based on:

- manuscript word count;
- editorial stage;
- complexity;
- citations, tables, footnotes, images, or other review load;
- exception severity;
- requested turnaround;
- publisher override.

The Author Workspace should not claim that a package is ready before release. While execution or QA is underway, the author-safe truth is the active work state, for example:

> The publishing team is completing the line editing and quality review of your manuscript.

## Marketing Attachment To The Lifecycle

Marketing should activate progressively from pipeline events and should use no-cost models wherever quality and governance allow. Human approval still controls public release.

Initial triggers:

- contract active;
- editorial path confirmed;
- Developmental Editing approved;
- title/subtitle finalized;
- cover concept approved;
- interior page count available;
- production proof approved;
- distribution submitted;
- retail listing verified;
- release date confirmed;
- release day;
- 7/30/60/90-day post-release reviews;
- backlist review date.

## Production And Distribution Branches

The same event-driven lifecycle must carry production, identifiers, distribution, audiobooks, and special formats.

Ebook generation should become a system-managed EPUB 3 pathway with EPUBCheck and device/render QA. Vellum remains a fallback, premium manual option, exception path, and output-comparison tool, not the mandatory runtime for ordinary text-forward ebooks.

ISBN assignment should be automated as a controlled publisher action after title, format, imprint, and publication posture are defined. The system must prevent duplicate assignment, incompatible format reuse, test-record assignment, premature assignment, and silent replacement of a distributed ISBN.

Distribution requires a governed route matrix by format, rights, territory, commercial terms, and available integration. Where no approved API exists, the system should produce a bounded manual submission package, record publisher confirmation, and start retailer validation automatically.

Audiobooks require a separate branch for narration strategy, agreements, chapter manifest, audio QA, mastering, distribution, and royalty ingestion. AI narration remains a separately approved pathway.

Children's and picture books require a distinct production profile with page/spread architecture, illustration brief, art direction, trim/bleed, fixed-layout ebook decision, accessibility treatment, and child-safety checks.

## Reverification Required

The remembered six-book distributor assumption must not be implemented until commercial and API terms are confirmed. The likely provider is PublishDrive, but current terms must be reverified before building any six-title rotation or API-dependent workflow.

## Implementation Boundary

The immediate production correction is not to ask Cody to keep starting stages. The correction is to commission the missing stage executors and cadence scheduler while showing every uncommissioned executor as bridge work until replaced.

Final operating target:

> Author approval -> next-stage runtime automatically executes -> QA and package creation complete -> manuscript-based cadence controls release -> author receives the next package.

That is the governing runtime architecture for editorial work, marketing, ebook creation, identifiers, production, distribution, audiobooks, royalties, and backlist operations.
